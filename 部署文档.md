# 部署文档

## 当前现象

服务器部署后，访问 `https://posterize.jototech.cn/api/auth/login` 时，Caddy 返回 `502`，日志中的关键信息为：

```text
dial tcp 127.0.0.1:8001: connect: connection refused
```

这表示：

1. 请求已经成功到达 Caddy。
2. Caddy 正在把 `/api/*` 转发到 `127.0.0.1:8001`。
3. 服务器本机的 `8001` 端口当前没有后端服务在监听。

结论：问题不在 Caddy 入口层，而在后端服务没有成功启动，或没有监听到预期端口。

## 代码中的部署预期

项目当前代码里，前后端部署关系如下：

- [Caddyfile](/c:/VM-workshop/威尔顿海报计划/Caddyfile)
  - `/api/*` 反向代理到 `127.0.0.1:8001`
- [deploy.sh](/c:/VM-workshop/威尔顿海报计划/deploy.sh)
  - 后端通过 `uvicorn main:app` 启动
  - 默认端口为 `8001`
  - Caddy 启动时也会把 `/api/*` 转发到 `${BACKEND_PORT}`，默认同样是 `8001`

因此，线上报错和仓库内的默认配置是一致的。

## 高概率原因

当前最可能是以下几种情况之一：

1. 后端进程没有启动成功
2. 后端启动后立刻退出
3. Python 虚拟环境或依赖缺失
4. 后端 `.env` 缺失或配置错误，导致启动失败
5. 后端没有监听在 `8001`
6. `uvicorn main:app` 的启动目录不对

## 推荐排查命令

登录服务器后，优先执行以下命令：

```bash
ss -ltnp | grep 8001
ps -ef | grep uvicorn
tail -n 200 /home/appadmin/aiposter/deploy-logs/backend.log
tail -n 200 /home/appadmin/aiposter/deploy-logs/caddy.log
```

### 结果解释

如果 `ss -ltnp | grep 8001` 没有输出：

- 基本可以确认后端没有在 `8001` 监听

如果 `backend.log` 中有报错：

- 优先按日志修复后端启动问题

如果 `uvicorn` 在运行，但不监听 `8001`：

- 检查 `BACKEND_PORT` 是否被改过
- 检查 Caddy 和后端端口是否一致

## 进一步排查命令

如果要手动确认项目目录和启动状态，可以继续执行：

```bash
cd /home/appadmin/aiposter
ls
ls deploy-logs
cat deploy-logs/Caddyfile
cat deploy-logs/backend.pid
ps -fp "$(cat deploy-logs/backend.pid)"
```

如果怀疑后端是启动即退出，可以手动重跑：

```bash
cd /home/appadmin/aiposter/backend
source .venv/bin/activate
python -V
pip show uvicorn
uvicorn main:app --host 0.0.0.0 --port 8001
```

如果手动启动时报错，终端里通常会直接看到真实原因。
